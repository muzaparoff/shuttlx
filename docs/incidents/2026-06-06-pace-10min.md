# Incident â€” Pace Pinned at 10'00 in Walk/Run

## Symptom

Users reported that pace in the walk/run workout timer always shows ~10'00 (ten minutes per kilometer) regardless of actual running speed, on both iPhone and Apple Watch.

## Root Cause

`currentPace` was a **misnomer**. Both `iPhoneWorkoutController` and `WatchWorkoutManager` computed it as cumulative average since workout start:

```swift
// iPhoneWorkoutController.swift line 467 (pre-fix)
self.currentPace = self.elapsedTime / self.totalDistance

// WatchWorkoutManager.swift line 971 (pre-fix)
currentPace = elapsedTime / distanceKm
```

The math is dimensionally correct (seconds Ă· km = s/km). But two compounding effects produced the 10'00 artifact:

1. **CMPedometer.distance warmup lag** â€” does not begin reporting for the first ~15â€“30 seconds of `startUpdates(from:)`. During that window `totalDistance == 0`, so the guard `if totalDistance > 0` skipped the assignment and `currentPace` stayed nil.
2. **First sample arrives skewed** â€” when distance starts arriving (around t=30s), the first reading is small (~0.05 km) while `elapsedTime` is already ~30s. That gives `30 / 0.05 = 600 s/km = exactly 10'00`. The cumulative average then **decays very slowly** â€” even after several minutes of fast running, the average is still dominated by that warmup ratio.
3. **GPS distance was discarded** â€” `CLLocationManager` updates were captured into `routePoints` (for the map) but never folded into `totalDistance`. So treadmill, indoor, or any GPS-dominant workout never recovered from the warmup spike either.

There was no hardcoded "10:00" or "600" fallback constant. The value was data-driven from the elapsed/distance ratio at workout start.

## Fix

Both targets now compute `currentPace` from a **sliding 30-second window** of `(elapsedTime, distance)` samples instead of a cumulative average from t=0:

```swift
// new shared state
private var paceWindowSamples: [(time: TimeInterval, distance: Double)] = []
private let paceWindowSec: TimeInterval = 30

// called each pedometer callback after totalDistance updates
paceWindowSamples.append((time: now, distance: totalDistance))
paceWindowSamples.removeAll { $0.time < now - paceWindowSec }
if let oldest = paceWindowSamples.first,
   now >= 20, totalDistance >= 0.05 {
    let dt = now - oldest.time
    let dDist = totalDistance - oldest.distance
    if dt >= 5, dDist >= 0.005 {
        currentPace = dt / dDist
    }
}
```

Guards applied:
- Workout must be â‰Ą 20s old (skips pedometer-warmup ratio window)
- Total distance must be â‰Ą 0.05 km
- Window must span â‰Ą 5s and â‰Ą 5 m before publishing a value
- Falls back to `currentPace = nil` ("â€”") when guards fail at startup; once published, retains the last value if the window momentarily can't compute a new one (e.g. brief stationary tick)

`paceWindowSamples` resets at every workout start and tear-down so deltas don't leak between sessions.
